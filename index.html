Step 1 — Idea to Spec · 5–15 min

Start with the spec.

The #1 reason vibe coding projects fall apart: jumping straight into building. A spec takes 10 minutes and saves hours of broken code.

A good spec answers 3 questions
What is this product?
Who is going to use it?
What will they actually do on it?
Then break it into phases. Phase 1 is core only. Everything else waits.
Use this prompt right now
"I want to build [your idea] using Next.js and Convex. Ask me questions to write a product spec with phases and screens."
Include your stack from the start — it shapes how the spec is written and means less back-and-forth later. Let it ask, answer honestly, then save the output.
Step 2 — Phases + Screens · 15–25 min

Plan before you build.

Break your spec into phases. Every screen gets listed before you write a single line of code. This stops scope creep before it starts.

Phase 1 — Core

Only what makes it work

Auth screen — sign up, log in
Main screen — the core interaction
Settings — bare minimum
If it's not critical to the core loop — it goes to Phase 2.
Phase 2+ — Nice to have

Everything else

Notifications, social features
Analytics, dashboards
Advanced settings, integrations
Ship Phase 1 first. Validate it works. Then build Phase 2.

Pick a design style first. Generate HTML mockups in 2–3 variations. Changing a mockup takes seconds — changing real code takes hours.

Step 4 — Build · 35–50 min

Build it properly.

Two rules that eliminate most beginner mistakes before they happen. Break either one and you will pay for it later.

At the start of every coding session
Paste your spec at the top of every new Claude Code session. It keeps the AI in context and reduces drift as the project grows.
Next.js Convex Convex Auth Vercel Claude Code
Two tips worth keeping in mind
1
Build backend functions in parallel with the frontend — it's easier to wire things up as you go than to retrofit later.
2
Avoid mock data where possible — real data connections from the start make testing more reliable and reduce surprises down the line.
Step 5 — Deploy + Iterate · 50–60 min

Deploy first. Then fix.

First pass is always ~75% there. Deploy early — it's the fastest way to verify your config is right, catch real-world issues, and share progress with anyone.

$ vercel --prodlive in ~30 seconds
Vague — slow to fix
"The timer is broken"
Specific — gets fixed fast
"The timer resets when I click pause. It should stop at the current time and resume from there when I click play again."
The more specific your prompt, the faster the fix. Describing the exact behavior gives the AI the context it needs to get it right.
