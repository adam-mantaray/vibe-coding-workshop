🖥️ Frontend

The stuff that makes vibe coding actually work

01

Component thinking

Break UI into reusable pieces. AI writes better code when you think in components, not pages.

02

State management

Know the difference between local state, server state, and global state. This is where most AI code falls apart.

03

Don't over-engineer

The AI will add Redux, Context, Zustand, and Jotai to the same project. You need to say "just useState."

⚙️ Backend

Where vibes go to die

01

Auth is not optional

Never roll your own. Use Clerk, Supabase Auth, or NextAuth. The AI will happily build you an insecure auth system.

02

Data model first

Design your schema BEFORE writing any code. Tables, relationships, access patterns. This is the skeleton.

03

API design matters

REST vs tRPC vs Convex functions. Pick one pattern and stick with it. AI mixes them freely — you shouldn't.

🗄️ Database

The thing nobody wants to think about

01

Schema = contract

Your database schema is a contract with your future self. Get it wrong and you'll rewrite everything.

02

Relations matter

Users → Orders → Items. One-to-many, many-to-many. If you can't draw it on a napkin, the AI can't build it.

03

Don't normalize everything

Sometimes denormalization is correct. Know when to duplicate data for speed. The AI will normalize into 47 tables.

The Knowledge Cheat Code

🧠

You don't need to write the code

You need to architect the solution and review the output.

🎯

Specificity > creativity

"Add a Stripe webhook handler that updates user.plan in Convex" beats "add payments."

🚨

Know what to reject

The AI will confidently write insecure code, wrong patterns, and fake APIs. Your CS brain is the filter.

🔄

Iterate, don't regenerate

Fix the specific issue. Don't throw away 500 lines because of one bug. That's vibe coding without knowledge.

Things AI Gets Wrong

That only knowledge catches

🔓

Security

Storing passwords in plain text. No input validation. SQL injection. Exposing API keys in frontend code.

💾

Data integrity

No transactions. Race conditions. Deleting without soft-delete. No backups strategy.

📦

Over-engineering

Microservices for a todo app. Kubernetes for 10 users. GraphQL when REST is fine.

🎭

Hallucinated APIs

Using packages that don't exist. Calling methods with wrong signatures. Referencing deprecated APIs.

The DARS Framework

How to prompt with knowledge

D

Define the data model

"I need users, projects, and tasks. Users have many projects. Projects have many tasks."

A

Architect the flow

"User signs up → creates project → adds tasks → marks complete. Dashboard shows stats."

R

Restrict the scope

"Only build the task CRUD for now. No notifications, no teams, no billing. Just tasks."

S

Specify the stack

"Use Next.js 15 app router, Convex, Tailwind, shadcn/ui. No other libraries."
